# Phase 4 — UI Design Contract (Chat Console)

> Visual and interaction contract for the Chat Console — the **lightest feature slice**, the **third consumer** of the Phase-3 SSE substrate. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **This contract INHERITS the Phase 1 design system AND the Phase 3 streaming visuals by reference and does NOT redefine them.** Phase 1 (`01-UI-SPEC.md`, approved) locks the dark-first operator theme, all color/typography/spacing tokens, the `--status-*` semantic layer, the icon/font stack, shadcn-on-Radix + Tailwind v4, and the cross-cutting primitives this phase MUST reuse: the **five-state pattern**, **app-wide toast (sonner)**, **raw-JSON viewer**, **copyable-id**, the **operator-context bar**, and the copywriting contract. Phase 3 (`03-UI-SPEC.md`, approved) is the **key precedent**: chat **REUSES its append/timeline frame-row renderer**, its **connection-state badge + machine** (`web/src/features/flow/timeline/connection.ts`), and its **streaming visuals** (per-frame icon+status color, mono, collapsed raw-JSON payload, the `loader`-spin live indicator). The sections below specify ONLY what is NEW for applying that system to the Chat Console's conversation surface. Where a token table is repeated, it is a verbatim re-statement of Phase 1 for the checker's six-dimension gate, not a new decision.
>
> **Aesthetic mandate (carried from Phases 1/3):** legibility and state-clarity over polish; the cardinal sin is **silent failure**. This phase carries forward the Phase-3 operator-critical distinction in a chat shape: **a Stop (operator-initiated, `closed` — neutral) must never read as an error (`error` step frame, red).** A stopped turn and a failed turn are different signals (D-05).
>
> **Source-grounding note (for the planner — CONTEXT flags the contract as research-gated; this researcher verified it against `../llm-agent-customer-support/internal/httpapi/httpapi.go` to ground the VISUAL contract; the planner still locks the schema/wiring before implementation):** `POST /chat/stream` emits SSE **named events** `step` / `done` / `error`, each a `StreamEnvelope{kind, answer, error}`. A **`step`** event's `kind` is the **agent StepKind** enum (`thought` · `action` · `observation` · `reflection` · `plan` · `final`; from `llm-agent-contract/agents/agent.go`) and `answer` carries the step's text content. **`done`** carries the final `answer` (the assistant's answer); **`error`** carries `error` (the failure string). **Session mechanic:** the request may carry `session_id`; if empty the server assigns a UUID and **returns it in the `X-Session-Id` response header** (also `session_id` in the sync `/chat` JSON body). The console reads that id, shows it via **CopyableId**, and **reuses it on the next turn** (grounds D-02/D-06). This matches the Phase-3 substrate exactly: named-event SSE-over-POST, a small terminal/non-terminal frame set, an id surfaced on first response.

- **Connection-state badge / "Stopped" marker chip** — ~20–22px chip height, `xs` (4px) internal padding; sub-grid chrome, not a content-rhythm token.
- **Step-trace gutter rail** — the Phase-3 1px neutral vertical rail (`#232A36`) connecting frame icons; reused inside the collapsed-expandable trace; a structural line, not a spacing token.

### Status / state colors (the operator semantic layer — NEW step-kind mapping; connection states reused from Phase 3)

These are functional signals, not accent. They reuse the Phase 1 `--status-*` tokens (and the Phase-3 connection mappings verbatim). Each pairs **color + text + icon** so state never relies on color alone (color-blind safe). **This is the core visual work of the phase.**

**(a) Per-step kind — the agent step trace (D-01).** Each `step` event's `kind` (the agent StepKind enum) gets a fixed icon + color in the step-trace gutter, reusing the Phase-3 frame-row renderer. Agent steps are **process/trace markers, not pass/fail status** — so they read as **neutral slate by default**, with the terminal `done`/`error` (NOT step kinds — they are separate SSE events) owning green/red. This keeps the trace calm and lets the terminal answer/error pop:

| SSE event / step `kind` | Color | Token | Icon (lucide) | Reads as |
|-------------------------|-------|-------|---------------|----------|
| `step` · `thought` | `#8A93A3` (slate muted) | `--status-unknown` | `brain` / `message-circle` | The agent reasoning |
| `step` · `plan` | `#8A93A3` (slate muted) | `--status-unknown` | `list-checks` | The agent planning steps |
| `step` · `action` | `#8A93A3` (slate muted) | `--status-unknown` | `wrench` / `zap` | The agent invoking a tool/action |
| `step` · `observation` | `#8A93A3` (slate muted) | `--status-unknown` | `eye` / `arrow-down-left` | The result the agent observed |
| `step` · `reflection` | `#8A93A3` (slate muted) | `--status-unknown` | `rotate-ccw` | The agent reflecting / self-correcting |
| `step` · `final` | `#8A93A3` (slate muted) | `--status-unknown` | `flag` | The agent's final step (precedes/accompanies `done`) |
| **`done`** (terminal — the answer) | `#22C55E` (green) | `--status-up` | `check-circle` | **Terminal success** — carries the final `answer`, rendered as the assistant bubble's collapsed answer (D-01) |
| **`error`** (terminal — the failure) | `#EF4444` (red) | `--status-down` | `x-circle` | **Terminal failure** — carries the `error` string, rendered RED in-bubble (D-01 stop-on-error) |

> **Step-kind color note:** the 6 step kinds are deliberately **all neutral slate** — they are trace breadcrumbs, not statuses, and an operator should not see a wall of green/amber/red mid-reasoning. The **only** color in the trace comes from the two terminal events: green `done`, red `error`. The icons (distinct per kind) carry the per-kind meaning without color. The **active/live step** (the current tail while streaming) gets the Phase-3 `loader`-spin treatment on its gutter icon (see the streaming indicator below) — same as a Phase-3 `node_started` live tail. **Unknown future step kinds** (the `kind` field is an open string — `StepEnvelope` from a `.loose()`-style parse) → fall back to a neutral `circle` icon + slate, never crash the renderer (mirrors Phase-3's loose-payload posture).

**(b) Streaming indicator — the live assistant bubble (D-01).** While a turn streams, the active assistant bubble shows:
- The **connection-state badge** "Streaming" (green `loader` spin / `radio`) — reused verbatim from Phase 3 (item (c)).
- A **`loader`-spin gutter icon on the live tail step** in the trace (the step currently arriving), identical to the Phase-3 live-node-tail treatment.
- (If no step has yet arrived between send and the first frame) a **muted "Thinking…" placeholder row** with a `loader` spin, so the bubble is never a bare blank while the stream opens (five-state: no silent blank).

**(c) Connection-state badge — the transport signal (D-05) — REUSED VERBATIM from Phase 3 (`ConnectionBadge`).** In/near the active assistant bubble. This is the signal an operator MUST NOT confuse with an `error` step:

| Connection state | Color | Token | Icon (lucide) | Label | Meaning (chat) |
|------------------|-------|-------|---------------|-------|----------------|
| **streaming** | `#22C55E` green | `--status-up` | `loader` (spin) / `radio` | **Streaming** | Stream open, steps arriving (live) |
| **closed** | `#8A93A3` slate muted | `--status-unknown` | `circle-check` / `square` | **Closed** | Terminal frame received (`done`/`error`) **OR operator pressed Stop** — stream cleanly ended. Closed is NEUTRAL — whether the turn *succeeded* is the terminal frame's job (green `done` answer vs red `error`), not the connection badge's. |
| **errored** | `#F59E0B` **amber** | `--status-degraded` | `unplug` / `wifi-off` | **Connection lost** | Transport drop, NO terminal frame and NO Stop — partial trace, recoverable. **Amber, not red** — a recoverable transport problem, not a turn failure. (Phase 5 adds `reconnecting`.) |

### The D-05 distinction (stated for the checker) — "Stopped" vs error vs connection-lost

This phase has **three** different "the turn didn't finish normally" signals; they MUST read as distinct (this is the phase's operator-critical contract, the chat analogue of Phase-3's `flow_err`-vs-transport-drop):

| Signal | What happened | Color | Where | Marker | Recovery |
|--------|---------------|-------|-------|--------|----------|
| **Stopped** (D-05) | **Operator** pressed Stop; the partial answer/steps stay | **muted slate** `#8A93A3` | a chip **in the assistant bubble** (below the partial trace/answer) | **"Stopped"** chip, `square` / `octagon` icon, muted | connection badge → **Closed** (neutral). This is **NOT an error.** Re-asking is a fresh **Send**, not a reconnect. |
| **Turn error** (`error` step) | The **agent/backend** failed mid-turn; `error` event arrived | **red** `#EF4444` | the red `error` frame **in the trace body** + an in-bubble error line | **"Failed — {error}."** red, `x-circle` | connection badge → **Closed** (the stream ended cleanly with an error *result*). No reconnect — re-asking is a fresh **Send**. |
| **Connection lost** | The **transport** dropped; no terminal frame, no Stop | **amber** `#F59E0B` | connection badge **near the bubble** + a muted line under the partial trace | **"Connection lost"** amber, `unplug` | connection badge → **Connection lost**; partial trace stays. (Phase 5 automates reconnect; this phase: re-Send.) |

> **The operator reading:** muted "Stopped" chip = *I* stopped it (benign). Red "Failed — …" in the body = the **agent** failed (a result). Amber "Connection lost" near the bubble = the **pipe** dropped (transport). Three colors (slate / red / amber), three locations, all distinct from one another and from a normal green-`done` turn. An operator must instantly tell "I stopped this" from "this failed" from "the connection died." Red is reserved for a genuine `error`; a Stop is **never** red.

`--status-down` / `Destructive` red is **reserved this phase for: the `error` terminal step frame + its in-bubble "Failed — …" line only.** A **Stop** is **muted slate** (operator-initiated, benign). A **dropped connection** is **amber** (recoverable). This honours the restrained Phase-1 red budget — red means "the turn genuinely failed."

### Destructive & confirmation copy

**No destructive actions this phase.** "New session" **clears the current transcript from the view** (D-06) — but there is **no history store**, nothing is deleted from any backend, and a fresh session id is assigned. Per the Phase-1 destructive bar (delete/disable require confirmation; benign view-resets do not), **"New session" fires directly with NO confirmation dialog** — it is a benign view reset, not a destructive backend operation (consistent with "no session-history store," CONTEXT D-06). Stop, Send, and toggling Stream/Sync are likewise non-destructive — fire directly. **There is no context gate** (customer-support is auth-none; unlike Phase-2 MEM-08).

> **Non-blocking note:** if the planner later finds operators lose in-progress work surprisingly, a lightweight "Start a new session? The current transcript will clear." inline confirm could be added — but CONTEXT D-06 chose a direct clear and recorded "keeping prior transcript visible" as explicitly *not chosen*. Default contract: **direct clear, no dialog.**

It specifies the interaction semantics the verified customer-support contract forces. **Schema/wiring is planner-locked against `../llm-agent-customer-support` (CONTEXT); this researcher grounded the visuals against that source — see the front-matter source-grounding note.**

### IC-1 — Single `/chat` route, single active session, "New session" (D-02/D-06)
- One route `/chat` (the third console nav target, no longer a placeholder). **No session-list / switcher / history** (no browse endpoint — CONTEXT; consistent with deferred memory sessions). v1 is **single-active-session**.
- **Session header (S1):** shows `SESSION` + the `session_id` via **CopyableId** (mono). The id is **server-assigned on the first turn** (read from the `X-Session-Id` SSE/JSON response header — verified in source) and **reused on every subsequent turn** (sent as `session_id` in the request, CHAT-02). Pre-first-turn: the muted "No session yet — send a message to start." treatment.
- **"New session" (D-06):** clears the conversation to the **empty state (S6)**, drops the in-view transcript (no history store), and resets the session id to "no session yet" (the *next* turn gets a fresh server-assigned id). One session at a time. Fires directly (benign view reset — no confirm; see Copywriting).

### IC-2 — Assistant turn = chat bubble + collapsible inline step trace, reusing the Phase-3 frame-row renderer (D-01) — THE KEYSTONE
- A turn appends a **user bubble (S3)** (the sent message, immutable) then an **assistant bubble (S2)**. Both **left-aligned, same Secondary surface, role-captioned** (see Color bubble note).
- The assistant bubble hosts a **collapsible inline "step trace"**: each arriving `step` event appends one **frame-row** via the **Phase-3 `TimelineView`/frame-row renderer** — gutter step-kind icon (per the step-kind color table) + step-kind label (sans Body) + the step content text (TEXT node, sans Body) + a **collapsed** Phase-1 raw-JSON viewer over the raw frame. The trace never reorders; it is an ordered log with the Phase-3 1px gutter rail.
- **Live → collapse-on-done (D-01):** while streaming, the trace is **expanded** (the operator watches steps arrive) with the live tail step `loader`-spinning and the **Streaming** badge. On the terminal **`done`**, the bubble renders the final **`answer`** (the `done.answer`, sans Body, the human-readable reply) and the step trace **collapses** to a **"{N} steps"** summary (re-expandable). The collapsed default keeps a finished transcript calm and scannable; the operator expands a turn's trace on demand. (Recommendation: the collapse is animated via the shadcn `collapsible` block; the answer is always visible, only the trace collapses.)
- Render ALL chat/step/answer/error strings as **TEXT nodes** (no `dangerouslySetInnerHTML`) — same XSS posture as Phases 2/3.

### IC-3 — One assistant bubble for streamed AND sync (D-03)
- **Send** (primary, accent, default) → imperative `openSseStream` (Phase-3 `lib/sse.ts`) on `POST /api/chat/stream` → the live step trace + streamed answer (IC-2).
- **Sync** (toggle → secondary path) → blocking `POST /api/chat` → `{answer, agent, session_id}`.
- **Both render into the SAME assistant bubble** (D-03 — one message surface, not two). The sync reply has **no step trace** (the sync endpoint returns only the final `answer`) → its bubble shows the `answer` directly (no "{N} steps" summary, or a disabled/empty trace affordance — recommendation: simply omit the trace for sync turns). In-flight treatment: **Send (stream)** → Streaming badge + live trace; **Send (sync)** → composer disables + a `loader`-spin **Thinking…** placeholder in the bubble until the response lands. A failed sync turn renders its `{error}` in that same bubble (red "Failed — {error}.") + the send-failure toast for a transport non-2xx.
- The sync/stream **toggle** lives **in the composer toolbar** (near Send) — see Registry for the block. Default **Stream** (D-03).

### IC-4 — Composer: multi-line, Enter-sends, disabled-while-streaming, Stop (D-04)
- **Multi-line** `textarea` (mono not required — prose). **Enter sends; Shift+Enter inserts a newline** (D-04). Empty/whitespace-only message → Send disabled (no empty turns).
- **While a turn streams:** the composer textarea + Send are **disabled** (D-04 — no concurrent turns, single active stream); the **Stop** button (neutral) is shown in place of / alongside Send.
- **Stop (D-04/D-05):** cancels the in-flight stream via the **Phase-3 abort seam** (abort the `fetch`/`AbortController`). On Stop, the partial answer + steps **stay** in the bubble (IC-2) with the muted **"Stopped"** chip; the composer **re-enables** for the next turn.

### IC-5 — Stop → `closed` (operator-initiated), NOT `errored` (D-05) — the operator-critical contract
- **Stop maps to the connection machine's `terminal` event** (`connection.ts`), NOT `transport-error` → the connection badge goes **Closed (neutral)**, exactly like a clean `done`/`error`. A Stop is an operator-initiated clean close, **never** the amber "Connection lost" errored state. (The Phase-3 terminal-then-error guard means a late `onError`/abort fallout after the operator-initiated terminal stays Closed — no flip to errored.)
- **The three not-finished-normally signals stay distinct (D-05):** **Stopped** = muted "Stopped" chip in-bubble + Closed badge (operator). **Turn error** = red `error` frame in the trace + "Failed — …" + Closed badge (agent result). **Connection lost** = amber badge + muted line (transport). See the D-05 distinction table in Color. **An operator must never confuse a Stop (benign, muted, I-did-it) with a failure (red) or a drop (amber).** This is the single most important visual contract in the phase.
- The connection-state machine is reused **as-is** and stays **extensible** (Phase 5 adds `reconnecting` between `streaming` and `errored` — no rewrite; do NOT hard-code a binary).

### IC-6 — Streams are IMPERATIVE, not TanStack Query (carried from Phase 3)
- Open on Send, append frames to local turn state, close on terminal (`done`/`error`) / Stop / unmount. Abort the upstream on unmount/Stop so customer-support doesn't keep running detached. **No surrounding REST** is required beyond the two send endpoints (no list/detail fetch) → TanStack Query is essentially unused this phase (the lightest slice). The planner mirrors the Phase-3 mock-`@microsoft/fetch-event-source` test harness (`web/src/test/mocks/fetch-event-source.ts`) for chat-frame tests.

## Non-Blocking Recommendations (genuinely-unspecified visuals — Claude's discretion per CONTEXT)

These are NOT locked by CONTEXT/REQUIREMENTS; recorded as recommendations for the planner/executor, none expanding scope:

1. **Bubble layout → left-aligned, same-surface, role-captioned + 2px left rail** (NOT consumer-chat left/right float). Operator-log aesthetic; assistant rail accent-blue (it carries state), user rail neutral. (Locked as the contract default in Color; recorded here as the discretion call.)
2. **Step-kind labels → mono affordance optional.** Keeping them sans Body (UI prose) is fine; mono is a minor recommendation only if the planner wants them to read as machine identifiers. Icons carry the per-kind meaning regardless.
3. **Sync/stream toggle → a shadcn `switch` (Stream default) OR a 2-segment `button` group.** Recommendation: a small **segmented button** (`button` blocks, already installed — no new registry block) labelled `Stream | Sync` reads more discoverably in an operator toolbar than a bare switch, and avoids adding `switch`. If the planner prefers a labelled `switch`, add the `switch` block (official registry, no vetting). Either satisfies D-03.
4. **Step-trace collapse default → collapsed once `done`, expanded while streaming** (locked in IC-2). A per-session "keep traces expanded" preference is out of scope (no settings surface this phase).
5. **"Thinking…" placeholder** appears only in the gap between Send and the first `step`/`done` frame; once any frame arrives it's replaced by the live trace. Keeps the bubble from ever being a bare blank (five-state: no silent blank).
6. **Auto-scroll** of the conversation column to the newest turn while streaming, reusing the Phase-3 auto-scroll-pause + "Jump to latest" affordance is **optional** this phase (a single-column transcript is shorter than a flow timeline). If adopted, reuse the Phase-3 affordance verbatim; if skipped, ensure the live tail stays in view on each new frame. Recommendation: reuse Phase-3 auto-scroll-pause for consistency.
