# Phase 3 — UI Design Contract (Flow Console)

> Visual and interaction contract for the Flow Console — the **keystone SSE phase**. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **This contract INHERITS the Phase 1 design system by reference and does NOT redefine it.** Phase 1 (`01-UI-SPEC.md`, status: approved) locks the dark-first operator theme, all color/typography/spacing tokens, the `--status-*` semantic layer, the icon/font stack, shadcn-on-Radix + Tailwind v4, and the cross-cutting primitives this phase MUST reuse: the **five-state pattern**, **app-wide toast (sonner)**, **raw-JSON viewer**, **copyable-id**, the **destructive-dialog pattern**, the **operator-context bar**, and the copywriting contract. Phase 2 (`02-UI-SPEC.md`, status: approved) is the **precedent** to mirror — the per-feature Interaction Contract (IC) format, the `features/` layout, the raw-JSON + zod editor surface, the react-table list, status badges, confirm weights, and URL-synced deep-linking. The sections below specify ONLY what is NEW for applying that system to the Flow Console's streaming surfaces. Where a token table is repeated, it is a verbatim re-statement of Phase 1 for the checker's six-dimension gate, not a new decision.
>
> **Aesthetic mandate (carried from Phase 1):** legibility and state-clarity over polish. The cardinal sin is **silent failure**. This phase adds the single most operator-critical distinction in the whole console: **a failed flow (`flow_err`, a real result) must never be confused with a dropped connection (transport error, a recoverable transport state)** — D-09. The connection-state badge and the terminal error frame are two different signals and must read as such.

## Spacing Scale

> Verbatim from Phase 1 (multiples of 4). No new tokens.

| Token | Value | Flow-Console usage |
|-------|-------|--------------------|
| xs | 4px | Frame icon-to-text gap, node-status chip padding, badge icon-to-label gap |
| sm | 8px | Timeline frame-row vertical padding (dense append-only log), table cell padding, node-status-strip item gap |
| md | 16px | Tab content padding, editor padding, run-trigger controls gap, card padding |
| lg | 24px | Gap between editor and run controls, gap between timeline and node-status strip, detail section breaks |
| xl | 32px | Empty-state vertical rhythm (no flows, no runs) |
| 2xl | 48px | Major detail-route region break (definition ↔ history) |
| 3xl | 64px | Page-level empty-state top offset |

**Density:** the live timeline is a **dense append-only log** — frame rows use `sm` (8px) vertical padding so a long run stays scannable; the node-status strip uses `sm` chip gaps. The editor and run-detail summary use `md` (16px). Flow detail is a **full route, not a drawer** (D-05 — a flow carries far more than a memory item); it uses the standard `md`/`lg` content rhythm with no fixed-width constraint.

Exceptions (carried from Phase 1, applied here):
- **Icon-only buttons** (copy-id, JSON collapse-toggle per frame, "jump to latest" affordance) render visual 16–20px but carry a **minimum 32px clickable target**.
- **Per-frame status icon** — 16px (`size-4`) lucide glyph in the timeline gutter; a frame-kind signal, not a spacing token.
- **Per-node status chip / connection-state badge** — ~20–22px chip height, `xs` (4px) internal padding; sub-grid chrome, not a content-rhythm token.
- **Timeline gutter / rail** — a 1px neutral vertical rail (`#232A36`) connecting frame icons; a structural line, not a spacing token.

## Color

> 60/30/10 split verbatim from Phase 1. The NEW work here is the **per-frame-kind**, **per-node-status**, and **connection-state** mappings onto the existing `--status-*` layer — these are functional state signals, NOT part of the accent budget.

| Role | Value | Flow-Console usage |
|------|-------|--------------------|
| Dominant (60%) | `#0B0E14` | Page background, table surface, timeline scroll surface |
| Secondary (30%) | `#151A23` (borders `#232A36`) | Cards, tab panels, editor surface, timeline panel + node-status strip surface, dialog surface, table header row |
| Accent (10%) | `#3B82F6` (blue 500) | See reserved-for list below |
| Destructive | `#EF4444` (red 500) | **Delete flow** button + the red destructive Delete dialog only |

Foreground: primary `#E6EAF0`, muted `#8A93A3` (timestamps, secondary metadata, skipped-node text).

**Accent (blue) reserved for — explicit list (Flow Console), never "all interactive elements":**
- The **single primary action per surface**: the primary **Run** button (the streamed/keystone trigger), the **Save** button in the editor, the **New flow** page button.
- **Focus rings** (app-wide, accessibility).
- The **`flow.id` / `run_id` copyable-id hover affordance** and inline links (run-history row → run sub-route).
- The **active tab indicator** (Definition / Runs) and the **active sort indicator** on a sorted table column header.
- The **"jump to latest" affordance** when the timeline is paused (a subtle accent pill so the operator's eye finds the resume control) — see IC-3.

Everything else stays neutral: the secondary **Run (sync)** button (NOT the primary action of its surface → neutral), Cancel/close, tab-bar chrome, table body, the timeline rail. **No blue on frame rows, status chips, badges, or the timeline body** — status color owns those, accent must not compete with it.

### Status / state colors (the operator semantic layer — NEW frame/node/connection mappings)

These are functional signals, not accent. They reuse the Phase 1 `--status-*` tokens. Each pairs **color + text + icon** so state never relies on color alone (color-blind safe). **This is the core visual work of the phase.**

**(a) Per-frame kind — the timeline (D-01).** Each of the 6 flowd frame kinds gets a fixed icon + color in the timeline gutter:

| Frame kind | Color | Token | Icon (lucide) | Reads as |
|------------|-------|-------|---------------|----------|
| `flow_started` | `#3B82F6` (blue, accent) *(see note)* | — | `play` | Run begins — a neutral lifecycle marker, not a status |
| `node_started` | `#8A93A3` (slate muted) | `--status-unknown` | `loader` (spin while that node is the live tail) / `circle-dashed` | Node is running / in-flight |
| `node_finished` | `#22C55E` (green) | `--status-up` | `check-circle` | Node completed OK |
| `node_skipped` | `#8A93A3` (slate muted) | `--status-unknown` | `circle-slash` / `skip-forward` | Node skipped (unmet edge condition); muted, not a failure |
| `flow_done` | `#22C55E` (green) | `--status-up` | `check-circle-2` | **Terminal success** — carries final `outputs` |
| `flow_err` | `#EF4444` (red) | `--status-down` | `x-circle` | **Terminal failure** — carries the `error` string (D-09) |

> **`flow_started` color note:** the two flow-lifecycle markers (`flow_started`, `flow_done`) are the only places blue appears in the timeline, and only as a *bookend lifecycle marker*, not interactive accent — this is an explicit, narrow exception to the "no blue in the timeline body" rule and does not expand the accent budget. If the checker prefers zero blue in the timeline, `flow_started` may instead use neutral slate `#8A93A3` with the `play` icon; node/terminal status colors are non-negotiable. **Recommendation: neutral slate for `flow_started`** to keep the accent budget pure and let green/red terminals own the eye.

**(b) Per-node status strip — in-place aggregation (D-01).** The compact strip lists each node once and updates IN PLACE as frames arrive (driven by the same reducer):

| Node status | Color | Token | Icon (lucide) | Treatment |
|-------------|-------|-------|---------------|-----------|
| **pending** (no `node_started` yet) | `#8A93A3` muted | `--status-unknown` | `circle-dashed` | Muted chip; node known from the flow but not yet reached |
| **running** (`node_started`, no `node_finished`) | `#8A93A3` muted | `--status-unknown` | `loader` (spin) | Spinner chip — the live node(s) |
| **done** (`node_finished`) | `#22C55E` green | `--status-up` | `check-circle` | Green chip |
| **skipped** (`node_skipped`) | `#8A93A3` muted | `--status-unknown` | `circle-slash` | Muted chip, reduced-emphasis (0.6 opacity) — inert branch |
| **errored** (the node whose failure produced `flow_err`, if attributable) | `#EF4444` red | `--status-down` | `x-circle` | Red chip — the failing node |

**(c) Connection-state badge — the transport signal (D-02/D-09).** In the timeline header. This is the signal that an operator MUST NOT confuse with `flow_err`:

| Connection state | Color | Token | Icon (lucide) | Label | Meaning |
|------------------|-------|-------|---------------|-------|---------|
| **streaming** | `#22C55E` green | `--status-up` | `loader` (spin) / `radio` | **Streaming** | Stream open, frames arriving (live) |
| **closed** | `#8A93A3` slate muted | `--status-unknown` | `circle-check` / `square` | **Closed** | Terminal frame received (`flow_done`/`flow_err`) — stream cleanly ended. Closed is NEUTRAL — the run finished; whether it *succeeded* is the terminal frame's job, not the connection badge's. |
| **errored** | `#F59E0B` **amber** | `--status-degraded` | `unplug` / `wifi-off` | **Connection lost** | Transport drop, NO terminal frame — partial timeline, recoverable via **Retry**. **Amber, not red** — it is a recoverable transport problem, not a flow failure. (Phase 5 adds `reconnecting` on top.) |

> **The D-09 distinction, stated for the checker:** a **`flow_err`** frame is **RED** and lives **IN the timeline body** (a real result — the run failed; keep the partial timeline visible). A **transport drop** flips the **connection badge** to **AMBER "Connection lost"** in the **header** (a recoverable transport state — offer Retry). Red-in-body = the flow failed. Amber-in-header = the pipe dropped. They are different colors in different places by design so an operator reads them instantly and never conflates them.

**(d) Run-status badge — history + run detail (D-07).** flowd `status ∈ {running, done, failed}`:

| Run status | Color | Token | Icon (lucide) | Label |
|------------|-------|-------|---------------|-------|
| **running** | `#8A93A3` muted | `--status-unknown` | `loader` (spin) | **Running** |
| **done** | `#22C55E` green | `--status-up` | `check-circle` | **Done** |
| **failed** | `#EF4444` red | `--status-down` | `x-circle` | **Failed** |

`--status-down` / `Destructive` red is **reserved for: `flow_err` terminal frames, `failed` run badges, the errored node chip, and the Delete-flow destructive dialog/button.** A dropped connection is **amber** (recoverable). `node_skipped` is **muted slate** (not an error). This keeps red meaning "the flow genuinely failed, or an irreversible delete," honoring the restrained Phase-1 budget.

| Secondary CTA — run (sync) | **Run (sync)** (neutral; blocking `POST /run` → same result surface, D-04) |
| Primary CTA — save flow | **Save** (editor; PUT/POST round-trip) |
| Secondary CTA — replay | **Replay** (on a completed run detail; re-streams persisted events → instant-fill, D-07) |
| Connection — manual retry | **Retry** (appears only when the connection badge is "Connection lost"; re-opens the stream — D-09) |
| Jump-to-latest (timeline paused) | **Jump to latest** (accent pill; resumes auto-scroll — D-03) |
| Inputs form label | **Inputs** (string→string key/value rows; flowd `runRequest.Inputs map[string]string` — values are strings only) |
| **Empty state — no flows** | Heading: **No flows yet.** Body: **Create a flow as JSON to run and observe it.** + primary button "New flow". |
| **Empty state — no runs** (flow has no run history) | Heading: **No runs yet.** Body: **Run this flow to see its execution history here.** |
| **Empty state — no events** (run exists, `GET /events` returns `200 {events:[]}`) | Heading: **No events recorded.** Body: **This run produced no events.** (NOT an error — empty events is a valid state; RESEARCH Pitfall 7.) |
| **Empty state — idle timeline** (before a run is triggered) | Muted: **Trigger a run to stream its event timeline.** (The timeline panel is never a bare blank — it states what to do.) |
| **Error state** (CRUD / run / list non-2xx) | **{HTTP status} from flowd — {error}.** + muted "Retry" + "View raw JSON" disclosure (Phase-1 raw-JSON viewer over the `{error}` body). Examples: `400 from flowd — flow compile: node "upper" unknown.` / `409 from flowd — flow "echo_chain" already exists.` / `404 from flowd — run "run_x" not found.` Never collapse the status/message into a generic string (BFF-04 passes the real flowd message through). |
| **Loading state** | **Loading…** with `loader` spin (flows list, flow GET, runs list, run GET, `/events` hydrate). Never a bare blank panel. |
| **Partial state** | Reuse the Phase-1 amber `alert-triangle` partial banner only if a sub-fetch degrades (e.g. run-detail summary loads but `/events` hydrate fails): **Showing partial data — {what's missing/why}.** |
| **`flow_err` terminal frame** (in the timeline body — D-09) | The red `flow_err` frame renders its `error` payload inline as the terminal row: **Flow failed — {error}.** (mono `error` string). The partial timeline above it STAYS visible. This is a *result*, not a transport error — no "Retry the connection" affordance (re-running is a fresh **Run**, not a reconnect). |
| **Connection lost** (transport drop — D-09) | Header badge: **Connection lost.** Below the (partial) timeline: muted **"Connection dropped before the run finished — {Retry} to re-open the stream."** Amber. This is the ONLY place a **Retry** (reconnect) appears. (Phase 5 automates this.) |
| **Toast — success** | Terse past tense, ≤ ~40 chars, auto-dismiss ~4s: `Flow created.` / `Flow saved.` / `Flow deleted.` / `Run started.` / `Run complete.` (sync). |

## Interaction Contracts (NEW behaviour — the load-bearing part of this phase)

> The checker validates copy/visuals/tokens above; this section is for the planner/executor. It specifies the interaction semantics the verified flowd contract (RESEARCH.md) forces.

### IC-1 — Flows list + flow detail are FULL ROUTES, not drawers (D-05/D-08)
- `/flows` — client-side `@tanstack/react-table` over `GET /flows` → `{flows:[FlowMeta]}`. **Columns:** `flow.id` (copyable-id, mono) · `name` (sans Body, falls back to id) · `created_at` / `updated_at` (mono Label timestamps). Row click → navigate to `/flows/{id}` (a real route, NOT a search-param drawer — a flow detail carries the editor + runs + run timelines, far more than a memory item). Five-state: loading / empty ("No flows yet" + New flow) / error / ready.
- `/flows/{id}` — flow-detail route with **tabs (Definition / Runs)**. Definition = the JSON editor + run trigger; Runs = the run-history table. **New flow** = `/flows/new` (or `/flows/{id}` with no id) → the same editor blank/templated.

### IC-2 — One editor, two modes; base64 round-trip + PUT id-match (D-05/D-06, RESEARCH-flagged)
- A single **route-hosted** mono JSON editor (reuses the Phase-2 raw-JSON + zod *pattern*; the planner decides whether to reuse the `EditorDrawer` component or a route variant — the PATTERN is locked: mono, inline parse errors, zod well-formedness gate). **Create mode** (new flow) → blank/templated flow IR. **Edit mode** → load `GET /flows/{id}`, **base64-decode the `json` field** to populate the editor (RESEARCH: `FlowRecord.JSON` is Go `[]byte` → base64 string; planner verifies empirically against one live GET — Open Q1).
- Validation ladder before Save: (1) `JSON.parse` → parse-error copy; (2) zod well-formedness of the envelope. Save (accent) disabled while unparseable. **flowd is the authoritative semantic validator** (`compileProbe` → 400) — surface its message via the error toast.
- On PUT, send `{name, flow}` with the **raw flow JSON** (NOT base64); **omit `id`** or set it equal to the URL id (RESEARCH Pitfall 4 — flowd 400s on a mismatched body id). On success → toast + refetch the flow. On create (POST) success → 201 → route to `/flows/{newId}`.

### IC-3 — Live timeline + node-status strip + auto-scroll-pause (D-01/D-03) — THE KEYSTONE
- **Append-only timeline (S5):** each arriving frame appends a row — gutter status icon (per-frame-kind color table) + frame-kind label (sans Body) + mono node name (when present) + a **collapsed** Phase-1 raw-JSON viewer over the frame payload. The timeline never reorders; it is an ordered log. A 1px neutral rail connects the gutter icons.
- **Per-node status strip (S5):** a compact strip listing each node once, updating IN PLACE (pending→running→done/err/skipped) per the per-node-status color table, fed by the **same reducer** over the **same frame stream** (D-01). Placed adjacent to (not interleaved with) the timeline.
- **Auto-scroll-pause (D-03):** while live and the operator is at the bottom, the timeline auto-scrolls to each new frame. On a **manual scroll-up**, auto-scroll **pauses** (the operator is inspecting); a **"Jump to latest"** accent pill appears (bottom-edge of the timeline) showing the operator is in paused/inspecting mode; clicking it scrolls to newest and **resumes** following. The paused-vs-following distinction is visible (the pill is present only while paused). No auto-scroll during replay/instant-fill (the whole run lands at once — D-07).
- Render ALL flowd event-payload strings as **TEXT nodes** (no `dangerouslySetInnerHTML`) — same XSS posture as Phase 2 (CONTEXT/RESEARCH).

### IC-4 — One result surface for streamed + sync runs (D-04)
- **Run** (primary, accent) → imperative `openSseStream` → the live timeline (IC-3). **Run (sync)** (secondary, neutral) → blocking `POST /flows/{id}/run` → `{outputs, run_id}`.
- **Both end in the SAME result surface:** the streamed run's terminal `flow_done` frame carries `outputs`; the sync run's response IS `outputs`. The sync result renders into the **same terminal-result panel** the streamed run ends on (a single outputs panel via the raw-JSON viewer), NOT a second parallel surface. In-flight treatment: **Run** → connection badge "Streaming" + per-node spinners; **Run (sync)** → the trigger disables + a `loader` spin on the result panel until the response lands. A failed sync run renders its `{error}` in that same panel (red) + an error toast.

### IC-5 — Streaming state machine + terminal close (D-02, RESEARCH Pitfall 6)
- Connection states: `idle → streaming → (closed | errored)`; designed to EXTEND (Phase 5 adds `reconnecting` between streaming and errored — do NOT hard-code a binary).
- **Both `flow_done` AND `flow_err` are terminal** — each flips the connection badge to **Closed** (neutral) and aborts the stream (closes the `AbortController`). `flow_err` ALSO renders its red terminal frame in the body (IC-6). A stuck "Streaming" badge after a terminal frame, or a leaked fetch, is a contract violation (RESEARCH Pitfall 6).
- A transport `onError` BEFORE any terminal frame → connection badge **errored** ("Connection lost", amber) + the Retry affordance (IC-6). Idle survival rides on nginx `proxy_read_timeout` (no heartbeat exists — RESEARCH); a long-silent node can drop the stream, and that surfaces here as "Connection lost", recoverable via Retry.

### IC-6 — `flow_err` vs transport drop are DISTINCT signals (D-09) — the operator-critical contract
- **`flow_err`** (a terminal frame, the run failed): RED frame **in the timeline body** with the `error` payload; partial timeline ABOVE it stays visible; connection badge goes **Closed** (the stream ended cleanly with a result). No reconnect affordance — re-running is a fresh **Run**.
- **Transport drop** (no terminal frame, the pipe died): connection badge goes **AMBER "Connection lost"** in the **header**; partial timeline stays visible; a manual **Retry** re-opens the stream (`/run/stream` again, or — for a run already created — recover via `/replay` / `/events` hydrate per the reducer's de-dup, RESEARCH). 
- **An operator must never confuse the two.** Different color (red vs amber), different location (body vs header), different recovery (re-run vs reconnect). This is the single most important visual contract in the phase.

### IC-7 — Run history + deep-linkable run sub-route + instant-fill replay (D-07/D-08)
- **Run history (S7):** on the flow-detail Runs tab, a client-side react-table over `GET /flows/{id}/runs` → `{runs:[RunMeta]}`. **Columns:** `run_id` (copyable-id, mono) · run-status badge (running/done/failed) · `started_at` / `finished_at` (mono Label; `finished_at` omitted while running → render "—"). Row click → navigate to `/flows/{id}/runs/{runId}`.
- **Run sub-route (S8):** `/flows/{id}/runs/{runId}` — a deep-linkable, shareable route where BOTH live runs and replays render. A live run gets its `run_id` from the `X-Run-ID` header as soon as flowd creates it, so the URL is shareable immediately. Browser-back → flow detail. Layout: run summary (status badge + timestamps + `GET /runs/{id}` `inputs`/`outputs`/`error` via raw-JSON viewer) + the **same timeline renderer** (S5/IC-3).
- **Replay = instant-fill (D-07):** opening a completed run hydrates `GET /runs/{id}/events` (or `POST /replay`) and **INSTANT-FILLS** the timeline through the **same reducer/renderer** — no artificial playback, no auto-scroll animation; the operator sees the whole run at once. De-dup on `(kind, node, ordinal)` (RESEARCH — `seq` is in `/events` but NOT in SSE frames; history is a clean prefix of live). Empty events (`200 {events:[]}`) → the "No events recorded" empty state, NOT an error (RESEARCH Pitfall 7).

These are NOT locked by CONTEXT/RESEARCH; recorded as recommendations for the planner/executor, none expanding scope:

1. **`flow_started` color → neutral slate** (not blue) to keep the accent budget pure; green `flow_done` / red `flow_err` own the terminal eye. (See Color note (a).)
