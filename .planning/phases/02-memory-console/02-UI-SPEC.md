# Phase 2 — UI Design Contract (Memory Console)

> Visual and interaction contract for the Memory Console. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **This contract INHERITS the Phase 1 design system by reference and does NOT redefine it.** Phase 1 (`01-UI-SPEC.md`, status: approved) locks the dark-first operator theme, all color/typography/spacing tokens, the icon/font stack, shadcn-on-Radix + Tailwind v4, and the cross-cutting primitives this phase MUST reuse: the **five-state pattern**, **app-wide toast (sonner)**, **raw-JSON viewer**, **copyable-id**, the **destructive-dialog pattern**, the **operator-context bar**, and the copywriting contract. The sections below specify ONLY what is NEW for applying that system to the Memory Console screens. Where a token table is repeated, it is for the checker's six-dimension gate and is a verbatim re-statement of Phase 1, not a new decision.
>
> **Aesthetic mandate (carried from Phase 1):** legibility and state-clarity over polish. The cardinal sin is **silent failure**. Memory adds one new first-class state to make unmistakable: the **`409 memory_conflict` (item changed under you)** OCC outcome — it is NOT a generic error.

## Spacing Scale

> Verbatim from Phase 1 (multiples of 4). No new tokens.

| Token | Value | Memory-Console usage |
|-------|-------|----------------------|
| xs | 4px | Badge icon-to-text gap, score-bar to number gap, tag chip padding |
| sm | 8px | Table cell vertical padding (dense rows), row-action menu item padding, drawer field-row gap |
| md | 16px | Search controls gap, drawer content padding, editor padding, dialog body padding |
| lg | 24px | Gap between search controls and results table, drawer section breaks |
| xl | 32px | Empty-state vertical rhythm (zero-results, unset-context gate) |
| 2xl | 48px | Major drawer region break (detail ↔ actions) |
| 3xl | 64px | Full-page gate top offset |

**Density:** dense operator table. Result rows use `sm` (8px) vertical cell padding to keep many hits scannable; the Advanced panel and drawer use `md` (16px). Drawer width: **fixed 480px** (desktop side-sheet) — wide enough for raw JSON + rendered fields without crowding the list behind.

Exceptions (carried from Phase 1, applied here):
- **Icon-only buttons** (row copy-id, JSON copy/collapse, row-action trigger) render visual 16–20px but carry a **minimum 32px clickable target**.
- **Pinned/disabled badges** — 20px row height, `xs` (4px) internal padding; a sub-grid chrome size, not a content-rhythm token.
- **Score bar** (if rendered) — 4px (`xs`) track height; a meter glyph, not a spacing token.

## Color

> 60/30/10 split verbatim from Phase 1. The NEW work here is the **status-color semantics for pinned/disabled badges** and the **in-flight / conflict treatments** — these live in the status layer, NOT the accent budget.

| Role | Value | Memory-Console usage |
|------|-------|----------------------|
| Dominant (60%) | `#0B0E14` | Page background, results-table surface, drawer scrim base |
| Secondary (30%) | `#151A23` (borders `#232A36`) | Drawer/sheet surface, Advanced panel card, editor surface, dialog surface, table header row |
| Accent (10%) | `#3B82F6` (blue 500) | See reserved-for list below |
| Destructive | `#EF4444` (red 500) | **Delete** button + the red destructive Delete dialog only |

Foreground: primary `#E6EAF0`, muted `#8A93A3` (timestamps, score captions, `source`/`category` secondary fields, disabled-row text).

**Accent (blue) reserved for — explicit list (Memory Console), never "all interactive elements":**
- The **single primary action per surface**: the **Search/Recall** button (search controls), the **Save** / **Submit** button in the editor drawer, the **"New record"** page button.
- **Focus rings** (app-wide, accessibility).
- The **`memory_id` copyable-id hover affordance** and any inline link (e.g. "open item").
- The **active sort indicator** on the sorted table column header (a small caret — accent so the operator sees which column drives the order).

Everything else stays neutral: row-action menu, pin/unpin/enable/disable buttons (neutral — they are not the primary action of their surface), `top_k` stepper, Advanced toggle, drawer close, secondary/Cancel buttons. **No blue on rows, badges, cards, or the table body.**

### Status / state colors (the operator semantic layer — NEW badge + state mappings)

These are functional signals, not accent. They reuse the Phase 1 `--status-*` tokens. Each pairs **color + text + icon** so state never relies on color alone (color-blind safe).

| State / element | Color | Token | Icon (lucide) | Treatment |
|-----------------|-------|-------|---------------|-----------|
| **Pinned** badge | `#F59E0B` (amber 500) | `--status-degraded` | `pin` | Filled amber chip + "PINNED" Label text. Amber = "operator-flagged / attention", not error. |
| **Disabled** badge | `#8A93A3` (slate muted) | `--status-unknown` | `circle-slash` | Muted outline chip + "DISABLED" Label text. The whole row also renders at reduced opacity (0.6) + muted text — a disabled item reads as inert. |
| **Active** (neither pinned nor disabled) | — | — | — | No badge. Absence of badge = normal. Do not add a "live"/"active" badge (badge spam). |
| **In-flight (pessimistic, D-11)** | `#8A93A3` (slate muted) | `--status-unknown` | `loader` (spin) | The acted row/drawer-action shows a spinner + the control disables + the row dims to 0.6 opacity during the round-trip. State flips ONLY after the backend confirms. |
| **OCC conflict (`409 memory_conflict`)** | `#F59E0B` (amber 500) | `--status-degraded` | `alert-triangle` | NOT red, NOT a generic error. Amber inline + toast: the item changed; auto-refetch is running; retry is offered. (See Interaction Contracts.) Amber because it is recoverable and expected, not a failure. |
| **Score** display | neutral (no status color) | — | — | Mono number `0.95`, two decimals, right-aligned, with an optional 4px neutral meter bar behind it. **Never** colored green/red — score is a rank, not a health signal; coloring it would steal the status budget. |

`--status-down` / `Destructive` red is **reserved for delete only** in this phase (the destructive dialog + the Delete button). A `409` is amber, a transport/5xx/4xx-other error is the standard Phase-1 red error state. This keeps red meaning "irreversible danger or hard failure," honoring the restrained budget.

## Copywriting Contract

> Tone, error formula, and toast formula are LOCKED by Phase 1 (terse, operator-grade, sentence case, errors carry `{status} from {service} — {upstream message}`). Below are the Memory-Console-specific strings, all conforming to that formula.

| Element | Copy |
|---------|------|
| Page title | **Memory** |
| Primary CTA — search | **Recall** (button label on the search controls; placeholder in query box: "Search memory…") |
| Primary CTA — new record | **New record** (page-level button; opens the editor drawer blank/templated, D-08) |
| Secondary CTA — more results | **Increase top-k** affordance label near the table when results == top_k cap reached: "Showing top {n}. Increase top-k to pull more (max 50)." |
| Advanced toggle | **Advanced** (collapsible; collapsed by default) — reveals score-threshold + metadata + pinned/disabled client-filters |
| Manual refresh | **Refresh search** (re-POSTs recall; never auto-runs after a mutation, D-09) |
| **Empty state** (valid recall, zero hits) | Heading: **No memory matched.** Body: **No items match "{query}" for this tenant/user. Try a broader query or raise top-k.** (Distinct from the unset-context gate and from an error.) |
| **Error state** (recall/get/mutation non-2xx, non-409) | **{HTTP status} from memory-gateway — {upstream message}.** + muted "Retry" + "View raw JSON" disclosure (Phase-1 raw-JSON viewer over the full `error` object). Example: `503 from memory-gateway — upstream unavailable.` |
| **Loading state** | **Loading…** with `loader` spin (recall in flight; drawer item fetch in flight). Never a bare blank panel. |
| **Partial state** | Reuse Phase-1 partial banner only if a sub-fetch degrades (e.g. drawer detail refetch-after-patch fails but the flag toggle succeeded): **Showing partial data — {what's stale/why}.** amber `alert-triangle`. |
| **OCC conflict (409)** — toast | **{Action} failed — 409: the item changed. Refreshing…** Amber. The console auto-refetches `GET /memory/items/{id}` to load the new `version`, then re-enables the action so the operator can retry against the fresh version. Includes "Copy error". (e.g. `Disable failed — 409: the item changed. Refreshing…`) |
| **Toast — success** | Terse past tense, ≤ ~40 chars, auto-dismiss ~4s: `Pinned.` / `Unpinned.` / `Disabled.` / `Enabled.` / `Record written.` / `Patched.` / `Deleted.` |
| **Toast — failure (non-409)** | Phase-1 formula: **{Action} failed — {HTTP status}: {upstream message}.** Persists until dismissed; "Copy error". Example: `Write failed — 400: content is required.` / `Patch failed — 400: expected_version is required.` |
| **Editor — parse error** (inline, before submit) | **Invalid JSON — {parser message at line/col}.** Red inline under the editor; Submit disabled while unparseable. |
| **Editor — schema error** (inline, zod, before submit) | **{field}: {rule}.** Examples: `content: required.` / `kind: must be one of working, episodic, semantic.` Red inline; Submit disabled while invalid. |
| **Editor — write mode placeholder** | Templated JSON pre-filled: `{ "kind": "semantic", "content": "" }` (the writable `record` object only — never `scope`/`idempotency_key`, the console assembles those). |
| **Editor — patch mode** | Pre-filled with the item's current patchable fields only (`content`, `category`, `tags`, `importance`); a note: **Only content, category, tags, importance are patchable. Use the lifecycle actions for pin/disable.** |

### Destructive & confirmation copy (D-10 — two weights)

| Action | Weight | Dialog copy |
|--------|--------|-------------|
| **Delete** | **RED destructive dialog** (Phase-1 pattern, first real use) | Title: **Delete memory item?** Body: **This permanently deletes `{memory_id}`. This cannot be undone.** Confirm button: **Delete** (Destructive red, repeats the verb). Cancel: neutral, **focused by default**. Pessimistic: row is removed only after `200` (D-11). |
| **Disable** | **NEUTRAL light confirm** (no red) | Title: **Disable this item?** Body: **Disabled items are excluded from recall until re-enabled. You can enable it again anytime.** Confirm button: **Disable** (neutral). Cancel default-focused. Reversible → no red (honors the destructive-red budget). |
| **Pin / Unpin / Enable** | **No confirm** | Non-destructive; fire directly with pessimistic in-flight treatment. |

## Interaction Contracts (NEW behaviour — the load-bearing part of this phase)

> The checker validates copy/visuals/tokens above; this section is for the planner/executor. It specifies the interaction semantics that the verified gateway contract (RESEARCH.md) forces.

### IC-1 — Results table is CLIENT-SIDE (D-03 forced by contract)
Recall returns a flat `hits[]` (≤ `top_k` ≤ 50), already sorted by `score` desc, with **no** server sort/offset/paging. Therefore:
- Sort, paginate, and pinned/disabled-filter are **all client-side** over the fetched hits (`@tanstack/react-table` `getSortedRowModel` / `getPaginationRowModel` / column filters).
- The ONLY server re-query lever is **`top_k`** (and `query` / `consistency_level`). Do **not** render server-pagination affordances. "See more" = raise `top_k` (max 50) → re-POST.
- **Columns (D-13/MEM-01):** `memory_id` (copyable-id, mono) · `score` (mono, 2dp, right-aligned, default sort desc) · `kind` (Label) · `source`/`category` (muted) · `content` (sans preview, truncate ~1 line, full in drawer) · pinned/disabled **badges** · timestamps (mono Label, if returned) · row-action menu (D-06). Row click (outside the action menu / copy-id) opens the drawer.

### IC-2 — Drawer synced to URL (D-04/D-05)
- Opening a row sets `?item={id}` (TanStack Router search param); the `sheet` slides over the list (list stays visible/dimmed behind). Reload reopens; browser-back closes it; the link is shareable.
- Drawer five-state: **loading** (item GET in flight) → **ready** (rendered fields + Phase-1 raw-JSON viewer + the full lifecycle action set + a "Patch" button opening the editor in patch mode). A `404 not_found` → **error** state inside the drawer (not empty).

### IC-3 — One editor, two modes (D-07/D-08)
- A single `sheet`-hosted mono JSON editor. **New record** (page button) → blank/templated `record` object (write mode). **Patch** (in drawer) → pre-filled with the item's patchable fields (patch mode).
- Validation ladder before Submit: (1) `JSON.parse` → parse-error copy; (2) zod (`writeRecordSchema` / `patchFieldsSchema`) → schema-error copy. Submit (accent) disabled until both pass. The editor edits ONLY the `record`/`patch` object — the console assembles `scope:{}`, `idempotency_key` (fresh UUID per submit), and `expected_version` around it.
- On success: write → toast + offer to open the new item (refetch GET for the body); patch → refetch GET, update drawer + the recall row's `version`. Never auto-re-run recall (D-09).

### IC-4 — Pessimistic lifecycle (D-11) + OCC version threading (RESEARCH, CRITICAL)
- Every mutation sends `expected_version` = the cached `version` (from the recall hit or GET). After success, the response's new `version` **replaces** the cached one.
- **In-flight visual:** the acted control disables + shows a `loader` spin; the row/drawer-action dims to 0.6 opacity. State flips ONLY after the backend confirms (no optimistic flip).
- **Reflect strategy (D-09 hybrid):** pin/unpin/disable/enable echo `{flag, version}` → `setQueryData` merge (no refetch). patch/write return only `{memory_id, version}` → **refetch `GET /memory/items/{id}`** for the authoritative body. delete returns `{deleted, version}` → splice the row out of cached recall data.

### IC-5 — `409 memory_conflict` is a first-class state (RESEARCH, NEW — not a generic error)
When any mutation returns `409 memory_conflict` (stale `expected_version`; `details: {memory_id, expected_version, current_version}`):
- **Amber, not red.** Toast: `{Action} failed — 409: the item changed. Refreshing…` (+ Copy error).
- The console **auto-refetches `GET /memory/items/{id}`** to load the new `version`, updates the cached item/row, and **re-enables** the action so the operator can retry against the fresh version. The drawer (if open) re-renders the new body so the operator sees what changed before retrying.
- This is the defined recovery — never a silent retry loop, never a dead-end red error.

### IC-6 — Confirms by reversibility (D-10) — see Destructive copy table above
Delete → red destructive dialog (pessimistic remove-after-200). Disable → neutral confirm. Pin/unpin/enable → no confirm.

### IC-7 — Full-page context gate (D-12) — see Unset-context copy above
Tenant or user unset → the whole route renders the Phase-1 unset-context empty-state; **no recall fires**, no controls render. Re-render the console only once both are set.
