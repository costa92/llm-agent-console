<objective>
Slice C-1 — the write/patch JSON editor (one editor, two modes) and the write/patch mutation layer with the D-09 refetch-after-write/patch reflect strategy and the first-class 409 OCC-conflict recovery. The editor edits only the operator's `record`/`patch` object; the console assembles `scope:{}`, a fresh `idempotency_key`, and `expected_version` around it. Validation runs a JSON.parse → zod ladder before Submit; mutations are pessimistic; a stale-version 409 surfaces as an amber recoverable state that auto-refetches and re-enables retry.

Purpose: After this slice an operator can write a new memory record and patch an existing one through a validated editor, with success/failure surfaced in toasts carrying the upstream message, and with stale-version conflicts handled visibly rather than silently. This establishes the `useMemoryMutations` hook + the `handle409Conflict` helper that plan 05's lifecycle actions extend.

Output: `EditorDrawer.tsx` (sheet, two modes, validation ladder), `useMemoryMutations.ts` (write + patch mutations + handle409Conflict), MemoryPage/ItemDrawer wiring to open the editor in each mode, and tests for validation, refetch-after, and the 409 recovery.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator JSON (textarea) → write/patch request | Operator-authored record/patch; zod is fast feedback, the gateway is the authoritative validator (V5) |
| Editor envelope assembly | scope:{}, idempotency_key, expected_version assembled by the console — never operator-edited (V4 scope confinement) |
| Lean mutation response → cache | Patch/write return {memory_id,version} only; body must be refetched, not trusted from the lean response |
| Stale version → 409 conflict | OCC gate: a stale expected_version is an expected, recoverable amber state, not a crash |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02C1-01 | Spoofing (confused deputy) | editor envelope scope/identity | mitigate [BLOCKING] | The editor edits only the record/patch object; scope:{} + idempotency_key + expected_version are assembled by client.ts/the hook, never from operator JSON. Acceptance grep asserts the editable template has no scope/idempotency_key/expected_version; identity rides only on X-Console-* |
| T-02C1-02 | Input validation (V5) | write/patch payload | mitigate | zod (writeRecordSchema/patchFieldsSchema) gates Submit (content non-empty, kind enum, at least one patch key, no non-patchable keys); the gateway remains authoritative; malformed JSON is rejected by the parse ladder before any request |
| T-02C1-03 | Tampering (lost update / OCC) | stale expected_version | mitigate [BLOCKING] | Every patch sends the cached version as expected_version; a 409 memory_conflict is surfaced as a first-class amber state (handle409Conflict): toast + auto-refetch GET item + re-enable retry — never a silent retry loop, never an optimistic flip-then-revert (D-11) |
| T-02C1-04 | Info disclosure | failure toast / error body | accept | SHELL-06 failure toast carries the real upstream status+message + Copy error; internal operator tool, intended legibility (BFF-04 passthrough) |
