# Phase 5 — UI Design Contract (Health & Hardening)

> Visual and interaction contract for the **cross-cutting hardening** phase (SHELL-02). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **This is a HARDENING phase. The design system is FULLY LOCKED across Phases 1–4 — this contract introduces NO new tokens, NO new components, NO new shadcn blocks, and NO new registries.** It inherits the Phase-1 design system (`01-UI-SPEC.md`, approved) verbatim, and **extends two already-executed visuals**: the Phase-1 shell **HealthDot** (`web/src/components/shell/HealthDot.tsx`) and the Phase-3/4 **connection-state machine + ConnectionBadge** (`web/src/features/flow/timeline/connection.ts`, `ConnectionBadge.tsx`). The genuinely-new contract is **three state additions** — live HealthDots, a `reconnecting` connection state, and the SSE disconnected/reconnecting overlay on stream views. Everything else is a citation, not a decision.
>
> **Aesthetic mandate (carried from Phase 1):** legibility and state-clarity over polish; the cardinal sin is **silent failure**. This phase makes the existing health dots LIVE and adds a recoverable **reconnecting** signal that must read as **distinct from** the already-locked streaming (green) / closed (neutral) / connection-lost (amber) / in-content-error (red) signals.

## Interaction Contracts (the load-bearing part — for the planner/executor)

> The checker validates the copy/visuals/tokens above. This section fixes the **interaction semantics** of the three extensions. All three are *additive* over executed code — extend, do NOT rewrite.

### IC-1 — Live HealthDots: one poll, four states, last-checked, stale-on-self-failure (D-01/D-02, SHELL-02)

- **Source:** the SPA polls **ONE** BFF endpoint `GET /api/health` (D-01) — a TanStack Query with `refetchInterval ≈ 15s` (D-02; health is a REST interval poll, **not** a stream). The response is `{ [service]: { status: up|down|degraded|unknown, lastChecked } }` for `memory` / `flow` / `chat`. The dot reads its `status` straight into the **existing** `HealthDot` `status` prop — **no visual change to the dot.**
- **State mapping (D-02):** `up` = probe 200; `down` = unreachable/non-2xx; `degraded` = probe OK but slow over the latency threshold (planner discretion); `unknown` = **pre-first-poll** OR **the `/api/health` call itself failed** (the aggregate poll errored → all dots go/stay `unknown` and **stale**, showing the last-known `lastChecked`). A failed aggregate poll must **never** blank the dots or flip them green — staleness is shown via `unknown` + "Checked … ago — health check unavailable."
- **Last-checked (D-01/D-02):** each dot carries a mono Label last-checked timestamp (relative, e.g. "8s ago") adjacent and/or in the tooltip — the EXISTING Phase-1 dot already reserves this; Phase 5 wires the live value. On stale/unknown the last-checked stays visible (proves the data is old, not absent).
- **Discretion (D-02, planner):** the slow→degraded latency threshold; whether the poll **pauses on a hidden tab** (visibility API). Recommendation below.
- **No new component, no new token.** This is `HealthDot.tsx` (executed) + a poll hook + the new BFF `/api/health` handler (BFF route, planner-owned).

### IC-2 — The `reconnecting` connection state: extend the machine, don't rewrite (D-03)

- **Machine extension:** add `reconnecting` to the `ConnState` union and the events needed (e.g. `reconnect-start` / `reconnect-success` / `reconnect-give-up`) to `connection.ts` — **purely additive**, exactly as the file's doc-comment anticipates ("Phase 5 can insert a 'reconnecting' state between 'streaming' and 'errored'"). Preserve the existing guards: **`terminal` still always wins → `closed`** (a `done`/`error` during a reconnect attempt settles to closed, never re-enters reconnecting); the terminal-then-error guard stays.
- **Transition shape (the contract, not the code):**
  - `streaming` + transport drop (no terminal frame) → **`reconnecting`** (start auto-retry; was previously the immediate `errored`).
  - `reconnecting` + a retry opens successfully → back to **`streaming`** (resume).
  - `reconnecting` + **terminal `done`/`error` arrives** (e.g. the reconnect lands mid-terminal, or a flow already completed server-side) → **`closed`** (stop — no reconnect storm; D-03 "STOPS on terminal").
  - `reconnecting` + **cap exhausted** (N attempts spent) → **`errored`** ("Connection lost", static amber, manual Retry only).
- **Backoff (D-03, planner/UI-SPEC discretion):** capped **exponential backoff + jitter**. Recommendation: **base ~1s, factor 2, max-delay ~30s, cap N ≈ 5 attempts**, full-jitter (delay = random(0, min(maxDelay, base·2^attempt))). The exact numbers are planner-tunable; the **cap is fixed/non-configurable** (no operator tuning UI — deferred). The `(n/N)` counter in the badge reflects `attempt / cap`.
- **Manual Retry always available:** throughout `reconnecting` (force an immediate attempt, resetting the backoff timer) AND after `errored` (re-open from the lost state). Retry reuses the existing Phase-3 `useRunStream.retry()` seam for flow / a fresh re-open for chat (see IC-3). Retry is a **neutral** control, never accent/red.
- **Badge:** `ConnectionBadge` gets the `reconnecting` arm (amber, spinning `loader`, label `Reconnecting (n/N)…`). The three existing arms are untouched.
- **A content error is NOT a drop:** a `flow_err` / chat `error` frame is a **terminal** → `closed` + red in-content frame — it must **NOT** trigger reconnect (D-03). Only a transport drop with **no** terminal frame enters `reconnecting`.

### IC-3 — Per-stream recovery: flow hydrate+de-dup vs chat re-open (D-03)

- **Flow runs:** a reconnect **hydrates `GET /runs/{id}/events` and de-dups on `(kind, node, ordinal)`** — the EXISTING Phase-3 `retry()` seam (`03-UI-SPEC.md` IC-5/IC-7). Auto-reconnect drives that same seam on a timer; the de-dup invariant guarantees no lost/duplicated frames when the stream resumes (history is a clean prefix of live). No fresh `/run/stream` for an already-created run.
- **Chat:** a reconnect **re-opens the stream** (`useChatStream`) — chat has no replay/de-dup (no `seq`/ordinals; `04-UI-SPEC.md` IC-6), so recovery is a re-open. (Planner: a chat re-open may re-emit steps; the chat reducer has no de-dup — acceptable per the Phase-4 contract, or the planner gates re-open behavior. This is a planner concern, not a visual one.)
- **Testability (CONTEXT specifics):** the reconnect layer must be unit-testable WITHOUT a live backend — reuse the Phase-3/4 fake SSE emitter (`web/src/test/mocks/fetch-event-source.ts`): script a drop → assert `reconnecting (1/N) → (2/N) …`, the cap, stop-on-terminal, and the flow de-dup-on-resume invariant.

### IC-4 — The SSE disconnected/reconnecting state is a TRANSIENT OVERLAY on the five states, not a replacement (D-04)

- The stream views (flow timeline, chat) already implement the five states (ready/streaming/error/…). The disconnected/reconnecting signal **composes ON TOP** as a transient transport overlay during a drop — it does **not** replace `ready` or the in-content `error`:
  - The **partial timeline/trace STAYS visible** (the already-arrived frames are `ready` content) while the **connection badge** flips to `Reconnecting (n/N)…` (amber) and a muted subline appears beneath: **"Connection dropped — reconnecting… (attempt n of N)."**
  - On reconnect success → the overlay clears, the badge returns to `Streaming`, new frames append to the same timeline (flow: de-duped).
  - On cap exhaustion → badge → `Connection lost` (static amber) + the existing Phase-3/4 muted "Connection dropped before the run/reply finished — Retry" line. Still an overlay; the partial content stays.
  - A terminal `done`/`error` ends it normally (badge `Closed`; a `flow_err`/chat `error` ALSO renders its red in-content frame — a *result*, never the reconnect overlay).
- **The four transport/result signals stay mutually distinct** (the phase's operator-critical contract, extending Phase-3 IC-6 / Phase-4 D-05): **green Streaming** (live) · **amber spinning Reconnecting (n/N)…** (dropped, re-trying) · **amber static Connection lost** (dropped, gave up — Retry) · **red in-content Failed — {error}** (the flow/turn genuinely failed, a result). Health adds a fifth, separate surface: **the shell dots** (up/degraded/down/unknown). An operator must read each instantly and never conflate "re-connecting" with "failed" with "the service is down."

### IC-5 — Five-state audit is a code-conformance retrofit, not a visual redesign (D-04)

- Audit every existing Memory/Flow/Chat view against the five-state contract (`01-UI-SPEC.md`); **fix gaps** found (a bare blank panel is the violation). Most surfaces already use `FiveStateWrapper` (Phases 2–4) — this is a targeted, evidence-driven retrofit. It produces **no new five-state visuals** — it makes existing surfaces conform to the already-locked contract. (This IC is for the planner/auditor; there is nothing here for the checker to validate beyond "no new tokens introduced.")

## Non-Blocking Recommendations (genuinely-unspecified visuals — Claude's discretion per CONTEXT)

These are NOT locked by CONTEXT/REQUIREMENTS; recorded as recommendations for the planner/executor, none expanding scope:

1. **Slow→degraded latency threshold (D-02 discretion):** recommend **degraded ≥ ~1s** probe round-trip (per service, server-side in the BFF handler), **down** on probe timeout (recommend ~3–5s per-service timeout, parallel probes). Tune against real latencies; keep the threshold server-side so the SPA just renders the returned `degraded`.
2. **Poll-pause on hidden tab (D-02 discretion):** recommend **pausing the 15s poll while the tab is hidden** (TanStack Query `refetchIntervalInBackground: false` is the default) and **refetching immediately on focus**, so a returning operator sees fresh health, not a 15s-stale dot. Show the last-checked timestamp so a just-revealed-but-not-yet-refetched dot still reads as possibly-stale.
3. **Reconnecting badge counter:** show `Reconnecting (n/N)…` with the live `n`; if the planner finds the counter noisy, a bare `Reconnecting…` (spinner only) is acceptable, but the **counter is recommended** — it tells the operator how close to giving up the client is.
4. **Backoff numbers (D-03 discretion):** recommend base 1s / factor 2 / max-delay 30s / cap N=5 / full-jitter (see IC-2). The cap is fixed (no operator-tuning UI — deferred per CONTEXT).
5. **Health-dot last-checked format:** recommend **relative** ("8s ago") updating on a light interval, with the **absolute** timestamp in the tooltip. Mono Label, per Phase 1.
6. **Manual Retry placement:** reuse the exact Phase-3/4 placement (a neutral control beneath the partial timeline/trace, beside/under the connection badge) for both `reconnecting` and `errored` — one consistent recovery affordance across both transport states.
