<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SPA → localStorage | Non-secret operator context (tenant/user/project/session IDs only); explicitly no auth tokens |
| SPA → BFF /api/config/env | Safe read-only endpoint; response must not contain secrets |
| SPA fetch → BFF /api/* | X-Console-* headers from localStorage context; operator auth token held in memory only (D-01) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | localStorage storing operator token | mitigate [BLOCKING] | D-01 mandates operator token is in-memory only; OperatorContextProvider stores ONLY {tenantId, userId, projectId, sessionId}; acceptance criteria grep: `grep -c 'operator_token\|Bearer' web/src/app/OperatorContextProvider.tsx` returns 0 |
| T-04-02 | Spoofing | SPA injects X-Console-* headers | accept | X-Console-* headers are non-secret operator context (tenant/user IDs); they are re-materialized server-side by the BFF Rewrite hook (Plan 03); the BFF strips all inbound X-*-Id headers and re-materializes from X-Console-* — so a malicious X-Console-Tenant only affects which tenant the operator's own session targets, which is the intended UX |
| T-04-03 | Information Disclosure | /api/config/env endpoint in SPA TopBar | mitigate | TopBar only displays the non-secret env/base URL fields from the response; acceptance criteria verify the BFF /api/config/env response never includes flowd_token or operator_token (Plan 03 task 2) |
| T-04-SC | Tampering | npm packages for shell components | mitigate | All packages in package.json verified via Package Legitimacy Audit in RESEARCH.md; no new packages added in this plan beyond what was installed in Plan 02 |
</threat_model>
