<task type="checkpoint:decision" gate="blocking">
  <decision>Production config injection: bind-mount config file (Option A) vs env-var override in config.Load() (Option B)?</decision>
  <context>
    The BFF config.Load() currently reads only a YAML file — no env-var override is implemented
    (the doc-comment mentions it but os.Getenv is never called). For prod secrets (flowd_token,
    operator_token), two approaches satisfy MVP:

    Option A — Bind-mount config file (no code change):
      - Operator creates config/config.prod.yaml on the host with real tokens (gitignored).
      - docker-compose.yml gains a volume mount + command override pointing at the prod file.
      - Pro: zero new Go code; exact same YAML format the operator already knows.
      - Con: requires managing a file on every deploy host.

    Option B — Env-var override in config.Load() (~15 lines of Go):
      - config.Load() reads FLOWD_TOKEN and OPERATOR_TOKEN from os.Getenv after YAML parse,
        overriding the YAML values when set.
      - docker-compose.yml gains two environment: entries (empty by default).
      - Pro: operator sets env vars (e.g. .env file or Docker secrets) — no file to manage.
      - Con: small code change; a .env file on the host has similar risks to a config file.

    For dev with empty tokens (the current committed config.dev.yaml), EITHER option requires
    no change — the BFF works as-is. This decision only affects prod/shared-dev deploys with
    real secrets.

    Research recommendation (06-RESEARCH GAP-4): both satisfy MVP; bind-mount (A) requires
    zero code change; env-var (B) is cleaner operator ergonomics.
  </context>
  <options>
    <option id="option-a">
      <name>Option A: Bind-mount config/config.prod.yaml</name>
      <pros>Zero new Go code. Familiar YAML format. Works immediately.</pros>
      <cons>Requires managing a file on each deploy host. File must be present at compose-up time.</cons>
    </option>
    <option id="option-b">
      <name>Option B: Env-var override in config.Load()</name>
      <pros>Clean operator ergonomics — set FLOWD_TOKEN / OPERATOR_TOKEN in shell or .env file.</pros>
      <cons>~15-line Go change in config/config.go + config_test.go update. One extra file to manage (.env).</cons>
    </option>
  </options>
  <resume-signal>Select: option-a or option-b. The executor will update deploy/docker-compose.yml (and optionally config/config.go) accordingly.</resume-signal>
</task>
